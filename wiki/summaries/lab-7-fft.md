# Lab 7 — Fast Fourier Transform

## TL;DR
Implement DIT FFT on the STM32F407 in both floating point and fixed point. The course provides bit-reversal and the skeleton loop — you fill in the **butterfly math**, the **scaling**, and the **twiddle factor update**. Test with a 4-pt vector, an 8-pt vector, and a 128-pt sinusoid generated in MATLAB.

## Tasks at a glance

1. **Fill in FILL_IN_BLANKs** in `fft()` and `fft_fixed_point()` in `main.c`. Five blanks per function:
   - Outer stage loop condition (bound is `num_stages`).
   - Butterfly count loop bound (`num_butterflies`).
   - Butterfly stride `i += num_pt_sub_DFT`.
   - Bottom index offset `i + num_butterflies`.
   - Temp/P/Q butterfly math (complex multiply, then $P = X[i] + \text{temp}$, $Q = X[i] - \text{temp}$, with optional $\times$ scale).
   - Twiddle update $\text{temp1.re} = U_{\text{re}}\cdot W[L-1]_{\text{re}} - U_{\text{im}}\cdot W[L-1]_{\text{im}}$.

2. **4-point test vector** (SIZE_FFT=4, LOG2_SIZE=2, no scaling):
   `{0.6, 0.3, -0.1, 0.3}`. Compare with hand calc. `shift_factor = 15` (Q15 fixed point).

3. **8-point test vector** (SIZE_FFT=8, LOG2_SIZE=3, no scaling):
   `{1, 2, 3, 4, 5, 6, 7, 8}`. Pick a `shift_factor` that keeps fixed-point result close to floating-point. Justify the choice.

4. **128-point sinusoid** (SIZE_FFT=128, LOG2_SIZE=7, scaling ON):
   Input comes from `sine_table.h` generated by `gen_sin_f.m`. Sinusoid frequency is **256 Hz** at $f_s = 8192$ Hz. $\Delta f = f_s/N = 64$ Hz $\to$ expect non-zero power at **bin 4** (and bin 124 by [[conjugate-symmetry]]). Then turn scaling OFF and observe overflow.

## Key implementation details

### The butterfly (floating point)
```c
temp.re = U.re * X[k].re - U.im * X[k].im;     // complex multiply
temp.im = U.re * X[k].im + U.im * X[k].re;
P.re = (X[i].re + temp.re) * scale;            // upper output
P.im = (X[i].im + temp.im) * scale;
Q.re = (X[i].re - temp.re) * scale;            // lower output
Q.im = (X[i].im - temp.im) * scale;
X[i] = P;  X[k] = Q;
```

### Twiddle update
```c
temp1.re = U.re * W[L-1].re - U.im * W[L-1].im;
U.im     = U.re * W[L-1].im + U.im * W[L-1].re;   // NOTE: must use OLD U.re here
U.re     = temp1.re;
```
The order matters: if you overwrite `U.re` first, the `U.im` update uses the wrong value. That's why `temp1` exists.

### Fixed point Q15
- `shift_factor = 15` means every value is stored as `value * 2^15`. So $1.0 \to 32768$, $0.5 \to 16384$, etc.
- After a multiply, the scale becomes $2^{30}$ — **right-shift by `shift_factor`** to renormalize.
- The twiddle update line in fixed point has the extra `>> shift_factor` for that reason.
- See [[fixed-point-arithmetic]] for detail.

### Scaling to prevent overflow
- At every stage, multiply $P$ and $Q$ by 0.5 (float) / shift right by 1 (fixed).
- After $\log_2 N$ stages, output has been shrunk by $(1/2)^{\log_2 N} = 1/N$.
- To recover, multiply by $N$ at the end. Code already does this when `SCALE == 1`.

## Questions raised
- Why does `shift_factor = 15` specifically? Because the STM32F407 has 32-bit int; Q15 means one sign bit + 15 integer bits + 16 fraction bits, with products fitting in 32 bits before renormalization. Covered in [[fixed-point-arithmetic]].
- What happens with scaling OFF on the 128-pt case? Overflow — large intermediate sums exceed the fixed-point range. That's the "why is scaling used" answer for the lab writeup.
